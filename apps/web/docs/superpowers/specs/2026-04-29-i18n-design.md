# i18n: Multi-language Support (en / es / pt / fr)

**Date:** 2026-04-29
**Status:** Approved, ready for implementation

## Goal

Add full-site language switching to YoBoss with browser auto-detection and
manual switching. Support four languages: English (default), Spanish,
Portuguese, French.

## Decisions

| # | Decision | Choice |
|---|----------|--------|
| 1 | Translation scope | Full site — every static UI string, public + authenticated routes |
| 2 | URL strategy | Cookie-only (no `/es/...` sub-paths). URLs unchanged. |
| 3 | Library | `next-intl` |
| 4 | Switcher placement | Top nav (logged-in) + Settings page row + Landing page header |
| 5 | Persistence | `NEXT_LOCALE` cookie only — no DB sync |
| 6 | Translation source | Claude (me) machine-translates all 4 locales during implementation |
| 7 | Email localization | Out of scope — emails stay English this round |
| 8 | AI response language | Match user's input language, independent of UI locale |

### Translation file shape

`messages/en.json` is the source of truth; other locales mirror its keys.

```json
{
  "common":   { "save": "Save", "cancel": "Cancel", "loading": "Loading…", ... },
  "nav":      { "notifications": "Notifications", "markAllRead": "Mark all read", ... },
  "landing":  { "hero": { "title": "...", "subtitle": "..." }, ... },
  "dashboard":{ ... },
  "goals":    { "list": {...}, "detail": {...}, "wizard": {...}, ... },
  "todos":    { ... },
  "team":     { ... },
  "workflows":{ ... },
  "settings": { ... },
  "account":  { ... },
  "admin":    { ... },
  "auth":     { ... },
  "languages":{ "en": "English", "es": "Español", "pt": "Português", "fr": "Français" }
}
```

Keys are namespaced by route or component family. Nesting is allowed but
kept shallow (≤3 levels).

### Locale detection flow

1. Middleware reads `NEXT_LOCALE` cookie.
2. If present and in `[en, es, pt, fr]` → use it.
3. Else parse `Accept-Language`, pick the highest-q match against the
   supported set; fall back to `en`.
4. Write the resolved locale back to cookie (1-year, `Lax`,
   `HttpOnly=false` so client switcher can read).
5. Continue to existing Supabase auth logic unchanged.

The cookie name `NEXT_LOCALE` is the next-intl convention.

### Server vs client usage

- **Server components / pages**:
  ```ts
  import { getTranslations } from "next-intl/server";
  const t = await getTranslations("goals.list");
  return <h1>{t("title")}</h1>;
  ```
- **Client components**:
  ```ts
  "use client";
  import { useTranslations } from "next-intl";
  const t = useTranslations("goals.list");
  ```
- **Date / number formatting**: switch isolated `toLocaleString()` calls
  (notifications, etc.) to `useFormatter()` so they respect the active
  locale.

### Switcher behavior

`<LanguageSwitcher>` is a client component, used in three places (with
optional `variant` prop for compact icon vs full row).

Click → calls server action:
```ts
"use server";
export async function setLocaleAction(locale: Locale) {
  const cookieStore = await cookies();
  cookieStore.set("NEXT_LOCALE", locale, { maxAge: 60 * 60 * 24 * 365, path: "/", sameSite: "lax" });
  revalidatePath("/", "layout");
}
```

### AI prompt change

For every system prompt in `src/lib/ai/*.ts` (goal-chat-prompt,
weekly-plan-chat, coach, persona, decompose, review, goal-detail-chat,
custom-tools, …), append:

> Respond in the same language the user writes in. Match their language
> naturally without announcing the switch.

UI locale is **not** passed to the AI. Detection happens implicitly from
the user's message.

## Out of scope

- Email template localization (Resend path, separate PR)
- DB schema changes / cross-device user-pref sync
- RTL languages
- Localized SEO (hreflang, per-locale sitemaps) — N/A under cookie-only
- Translation of user-generated content (goal titles, todo names,
  user-named agents)
- Translation of dev-facing strings (console logs, error stack traces)
- Translation of third-party brand names (Stripe, Supabase, Claude, …)

## Implementation order

A single PR, but commits sequenced for review:

1. **Infra** — install `next-intl`, add `i18n/`, wire middleware,
   dynamic `<html lang>`, four empty `messages/*.json`, root layout
   provider.
2. **Switcher** — component + insert in top-nav, settings page, landing
   header. At this point switcher works but most strings still hardcoded.
3. **Public pages** — extract + translate landing, pricing, privacy, terms,
   auth modal.
4. **App pages** — extract + translate dashboard, goals, todos, team,
   workflows, settings, account, admin.
5. **AI prompts** — append language-matching instruction.

Each commit must build green (`npx tsc --noEmit` + `npx next build`).

## Verification

Automated:
- `cd apps/web && npx tsc --noEmit` clean
- `cd apps/web && npx next build` clean

Manual (user runs on local dev server):
- First visit with browser language Spanish → site loads in Spanish
- Switch to French via top-nav → entire site re-renders in French, cookie
  set, persists across navigation
- Switch via Settings page → same effect
- Switch on landing (logged out) → also works
- Send AI agent a message in Chinese while UI is in French → agent replies
  in Chinese
- All four locales render without missing-key warnings in console

## Risks / open issues

- **Translation quality**: machine-translated strings will have rough edges.
  Acceptable for v1; flag for native-speaker pass before public launch.
- **Long strings overflow layouts**: French/Portuguese tend to expand 15–30%
  vs English. Watch for button/nav truncation during manual QA; fix with
  `truncate` or layout tweaks.
- **Existing `toLocaleString` calls**: a quick grep will be needed to catch
  them all and route through `useFormatter()`.
